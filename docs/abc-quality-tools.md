# ABC Normalize, Validate, And Repair Design

This document designs a future Studio feature for helping users clean up ABC generated by AI tools. It is intentionally a design document only; no checker implementation exists yet.

## Product Goal

AI tools often produce ABC that is close to useful but has small structural mistakes: missing headers, malformed fields, uneven bars, accidental extra prose, repeated code fences, or voice labels that do not line up. ChatMusic should help users understand and fix those issues before rendering, copying, or exporting the score.

The feature should feel like a local score quality assistant, not a strict compiler. It should preserve user intent, explain problems clearly, and only auto-repair changes that are predictable.

## Recommendation

Build this ourselves as a small rule engine on top of abcjs parsing instead of trying to write a full ABC parser from scratch.

The practical architecture is:

- Use **abcjs** as the ground-truth parser/renderability check.
- Add a **ChatMusic ABC lint layer** for AI-specific source quality checks.
- Add a **normalizer** for safe formatting and field cleanup.
- Add a **repair planner** that proposes or applies narrow, reversible fixes.

This is moderate complexity if the scope stays focused. It becomes high complexity only if we try to fully understand all ABC dialects, engraving details, tuplets, lyrics, repeats, and semantic bar duration validation across every meter.

## abcjs Warning Mechanism

abcjs already exposes a useful, tolerant warning path:

- `abcjs.parseOnly(source, { generate_warnings: true })` returns tune objects.
- Each tune can include `warnings?: string[]`.
- Warning strings often contain a source hint such as `Music Line:4:5` and an HTML-underlined token.
- abcjs tends to keep parsing instead of throwing, so many malformed snippets still produce a tune object.

This is enough for a first feedback-focused checker. It is not a strict ABC standard validator. For example, some incomplete or questionable input may parse with no warnings, while unknown symbols can appear in `warnings` as ignored characters.

The first implementation should treat abcjs warnings as the primary signal, sanitize their HTML snippets for display, and avoid promising complete standard compliance.

## Non-Goals

- Do not implement a complete replacement for abcjs parsing.
- Do not promise professional engraving validation.
- Do not auto-rewrite musical content when the intended rhythm, pitch, or voice structure is ambiguous.
- Do not use a remote LLM or server-side validator.
- Do not validate copyrighted source provenance.

## User Experience

Studio can expose this as a compact quality panel near the editor tools:

- **Validate**: scans the current ABC and reports issues.
- **Normalize**: applies safe formatting changes.
- **Repair**: applies only high-confidence fixes, with a summary of what changed.
- **Copy for AI**: copies the cleaned ABC with a short instruction wrapper for AI tools.

Issue messages should be grouped by severity:

- **Error**: likely prevents parsing or rendering.
- **Warning**: may render, but likely indicates a musical or structural problem.
- **Suggestion**: style, readability, or AI-exchange improvement.

Each issue should include:

- A short title.
- A plain-language explanation.
- A line number when possible.
- A suggested action.
- Whether it can be automatically fixed.

## Example Diagnostics

### abcjs Parser Warning

Severity: Warning  
Message: `Unknown character ignored.`  
Guidance: `abcjs ignored part of the source while parsing. Check the highlighted token before exporting or copying to AI.`  
Auto-repair: No by default.

This should be generated from abcjs `warnings` whenever possible rather than from a hand-written grammar rule.

### Missing Required Header

Severity: Error  
Message: `Missing K: key field.`  
Guidance: `ABC tunes need a K: field before note content. Add K:C if the key is unknown.`  
Auto-repair: Add `K:C` before the first detected music line.

### Code Fence Included

Severity: Warning  
Message: `Markdown code fence found inside ABC source.`  
Guidance: `Remove triple backticks before rendering or exporting.`  
Auto-repair: Remove leading and trailing fences when they wrap the whole source.

### Prose Around ABC

Severity: Warning  
Message: `Text outside ABC fields and music lines was found.`  
Guidance: `AI responses sometimes include explanations inside the copied source. Keep only ABC fields and notation lines.`  
Auto-repair: Only if the ABC block boundaries are obvious.

### Duplicate Title Fields

Severity: Suggestion  
Message: `Multiple T: title fields found.`  
Guidance: `ABC allows multiple titles, but AI often duplicates them accidentally. Keep them only if intentional.`  
Auto-repair: No by default.

### Voice Used Before Declaration

Severity: Warning  
Message: `Voice V:2 appears in notation before a matching voice declaration.`  
Guidance: `Declare voices near the header, for example V:1 and V:2, before voice-specific music lines.`  
Auto-repair: Add a simple `V:` declaration only when voice IDs are unambiguous.

### Suspicious Bar Length

Severity: Warning  
Message: `This measure may not match the current meter.`  
Guidance: `Check note lengths around this bar. AI output often drops duration suffixes or barlines.`  
Auto-repair: No. Rhythm repair is usually ambiguous.

## Normalization Scope

Safe normalization can be implemented before deeper validation:

- Normalize line endings to `\n`.
- Trim trailing whitespace.
- Remove wrapping Markdown fences.
- Ensure a single final newline for exported `.abc` files.
- Separate header fields and notation blocks with predictable blank lines.
- Keep known ABC field order when reordering is safe: `X`, `T`, `C`, `M`, `L`, `Q`, `K`, then voice declarations.
- Preserve comments and unknown fields unless a user explicitly asks for aggressive cleanup.

Normalization should avoid changing note tokens unless there is a precise rule and a test.

## Validation Scope

Validation can be layered in three phases.

### Phase 1: Source Hygiene

Low complexity. Good first implementation target.

- Empty source.
- Markdown fences.
- Missing `X:`, `T:`, or `K:` fields.
- Header fields after music has started.
- Multiple tunes in one source.
- Suspicious prose lines.
- Unsupported file/import remnants.

### Phase 2: Parser Integration

Moderate complexity.

- Run `abcjs.parseOnly()` and collect parse warnings when available.
- Confirm at least one tune object is produced.
- Confirm `abcjs.renderAbc()` can render into a detached element in tests.
- Map parser warnings back to source lines when abcjs provides character offsets or when a token can be located safely.

For the first release, Phase 2 can be very small: parse with `generate_warnings`, display sanitized warning text, and show a general "abcjs parsed this with warnings" status. It does not need deep line mapping on day one.

### Phase 3: Musical Structure Checks

Moderate to high complexity. Add carefully.

- Voice declarations and voice usage consistency.
- Basic bar duration checks for simple meters and simple note lengths.
- Unclosed chords, broken rhythm markers, tuplets, slurs, ties, and repeat brackets.
- Detect very long lines that hurt AI exchange readability.

Bar duration checks should start with simple monophonic lines and clearly report uncertainty for complex constructs.

## Repair Scope

Auto-repair should be conservative. Suggested rule:

Only repair when the change is syntactic, reversible, and unlikely to alter musical intent.

Good repair candidates:

- Strip wrapping Markdown fences.
- Trim prose before the first `X:` or after the final obvious notation line.
- Add `X:1` when missing.
- Add `K:C` when missing and no key can be inferred.
- Normalize line endings and whitespace.
- Move obvious header fields above notation if doing so preserves relative field order.

Poor repair candidates:

- Changing note durations to make bars add up.
- Guessing missing accidentals.
- Splitting or merging voices.
- Rewriting tuplets, repeats, ornaments, or lyrics.
- Inferring key from melody.

## Proposed Module Shape

Future implementation should keep logic isolated and testable:

```text
src/shared/abc-quality/
  diagnostics.ts     Types for issue severity, source range, and repair actions.
  normalize.ts       Safe source normalization.
  validate.ts        Rule runner and abcjs parser integration.
  repair.ts          High-confidence repair planner/applier.
  ai-copy.ts         Optional prompt wrapper for AI exchange.
```

Studio should call these modules and render results. The shared modules should not depend on DOM APIs except where abcjs requires a detached rendering target for parser/render checks.

## Testing Strategy

Every rule should have focused unit tests with small ABC fixtures. Tests should include both the diagnostic and the repaired output when auto-repair is supported.

Suggested fixture categories:

- Clean minimal ABC.
- AI response with Markdown fences.
- Missing `K:` field.
- Header mixed into notation.
- Multi-voice score.
- Simple bar-length mismatch.
- Complex ABC that should not be auto-repaired.

Run the final repaired output through abcjs parsing whenever possible.

## Complexity Estimate

- Phase 1 source hygiene and safe normalization: **small to medium**.
- Phase 2 abcjs parser integration and line mapping: **medium**.
- Phase 3 musical structure checks: **medium to large**, depending on how many ABC constructs are supported.
- Fully automatic musical repair: **large and not recommended** for the current product scope.

The best first release is a thin Phase 2 parse check plus a few Phase 1 source hygiene checks. That would already catch common AI mistakes without pretending to be a complete ABC compiler.

## Minimal Feedback-First Version

If we want to move quickly, the first implementation can be limited to:

- Run `abcjs.parseOnly(source, { generate_warnings: true })`.
- Display sanitized `tune.warnings` as warnings.
- Report "No abcjs parser warnings" when abcjs is satisfied.
- Detect wrapping Markdown fences and prose outside an obvious ABC block.
- Offer only one auto-fix: strip wrapping Markdown fences.

This gives users immediate feedback when AI output contains obvious parse problems while keeping the feature small and honest.
